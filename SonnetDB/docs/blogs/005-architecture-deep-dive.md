## 架构深度解析：SonnetDB 的写入与查询路径

理解一个数据库的内部架构，是高效使用和深度定制的前提。SonnetDB 虽然上手简单，但其内部实现了一套精妙的存储与查询引擎。本文将深入剖析 SonnetDB 的核心架构，包括写入路径、查询路径和数据压缩（Compaction）机制。

### 整体架构概览

SonnetDB 的存储引擎采用 **LSM-Tree（Log-Structured Merge-Tree）** 风格的架构设计，包含三个核心组件：**WAL（Write-Ahead Log，预写日志）**、**MemTable（内存表）** 和 **Segment（持久化段文件）**。这种设计在写入性能和读取性能之间取得了良好的平衡，特别适合时序数据写多读少的场景。

此外，SonnetDB 还包含了查询引擎、索引管理器和 HTTP API 层，共同构成了完整的数据库系统。

### 写入路径（Write Path）

当您执行一条 `INSERT` 语句时，数据会经历以下三个阶段：

**第一阶段：写入 WAL**。每个写入请求首先被追加到 WAL 文件中。WAL 是一个顺序追加的日志文件，确保即使在写入过程中系统崩溃，数据库也能在重启后通过回放 WAL 恢复数据。这种"先写日志、后写数据"的策略是数据库系统保证数据持久性的经典做法。

**第二阶段：写入 MemTable**。WAL 写入成功后，数据被写入内存中的 MemTable。MemTable 是一个有序的数据结构（通常实现为跳表或有序树），数据按照时间戳和标签键排序，为后续的查询优化做好准备。

**第三阶段：刷盘为 Segment**。当 MemTable 的大小达到阈值（通常为几 MB 到几十 MB）时，SonnetDB 会将其冻结为不可变的 MemTable，然后在后台异步写入磁盘，形成一个 Segment 文件。Segment 是磁盘上的只读数据文件，采用 SonnetDB 自定义的列式存储格式，针对时序数据进行了压缩优化。

### 查询路径（Query Path）

当执行 `SELECT` 查询时，SonnetDB 需要从多个数据源合并读取：

1. **MemTable 扫描**：首先检查当前活跃的 MemTable 和所有冻结的 MemTable，这保证了最新写入的数据能够被立即查询到。
2. **Segment 扫描**：然后扫描所有相关的 Segment 文件。SonnetDB 会利用时间范围过滤、标签索引等机制，快速跳过不相关的 Segment，减少 I/O 开销。
3. **结果合并**：将 MemTable 和 Segment 的查询结果按照时间戳顺序进行合并，去除重复数据，应用 `WHERE` 过滤条件和聚合函数，最终返回给客户端。

这个过程中，SonnetDB 使用了一种称为 **"时间分区"** 的优化策略：数据按照时间范围自动分区到不同的 Segment 中，查询时可以通过时间谓词快速定位到目标分区，从而显著提升查询性能。

### 压缩机制（Compaction）

随着数据的持续写入，磁盘上会积累越来越多的 Segment 文件，这会导致查询性能下降（因为需要打开和扫描的文件变多了）。为了解决这个问题，SonnetDB 的后台 Compaction 线程会定期执行以下操作：

- **合并小文件**：将多个小 Segment 合并成一个大 Segment。
- **数据清理**：删除已标记为过期的数据（TTL 过期）。
- **去重**：当多个 Segment 中存在相同时间戳的数据时，保留最新的版本。

Compaction 过程在后台异步运行，对前台查询和写入的影响极小。SonnetDB 使用了一种层次化的 Compaction 策略，类似于 RocksDB 的 Leveled Compaction，在写入放放大和读放大之间取得了良好的平衡。

### 架构示意图

```
写入请求 → [WAL] → [MemTable] → [冻结 MemTable] → [Segment（磁盘）]
                                  ↓                      ↓
查询请求 → [MemTable 扫描] + [Segment 扫描] → [结果合并] → [查询结果]
                                  ↓
                           [Compaction 线程] → 合并/清理/去重
```

通过这种架构设计，SonnetDB 在保持写入吞吐量的同时，提供了可预测的查询性能。理解这条写入和查询路径，可以帮助您在实际使用中做出更好的数据模型设计和性能调优决策。
